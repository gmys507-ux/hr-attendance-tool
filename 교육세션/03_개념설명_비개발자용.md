# 🧠 비개발자용 핵심 개념 설명 — 5가지

> 진행 원칙: 각 개념을 **(1) 한 줄 정의 → (2) 일상 비유 → (3) HR 근태 사례 연결 → (4) 왜 알아야 하나** 4박자로.
> 코드는 화면에 띄우되 "읽으라"가 아니라 **"이게 그 레고 블록이에요"** 수준으로만 가리킨다.

---

## 개념 1. 프로그래밍 언어 / 파이썬(Python)

**한 줄 정의** — 컴퓨터에게 "이 순서대로 해"라고 시키는 약속된 언어. 파이썬은 그중 사람 말과 가장 비슷한, 가장 쉬운 축.

**일상 비유** — 외국에서 일을 시키려면 그 나라 말이 필요하죠. 컴퓨터에게 일 시키는 "외국어"가 프로그래밍 언어이고, 파이썬은 "단어도 쉽고 존댓말도 필요 없는, 제일 배우기 편한 외국어"예요. 영어가 글로벌 공용어이듯, 파이썬은 자동화·데이터·AI의 공용어.

**HR 사례 연결** — 근태정리 도구는 처음부터 끝까지 파이썬으로 돕니다. "엑셀 43명 데이터를 읽어 → 두 시스템(원티드·FLEX)을 합치고 → 초과근무를 계산해 → 새 엑셀로 저장해"라는 매월 같은 지시를, 사람 대신 파이썬이 30초에 수행.

**왜 알아야 하나** — "코딩 = 어렵고 나랑 무관"이라는 벽을 깨려고. 파이썬을 외울 필요는 없습니다(그건 AI가 합니다). 다만 "내 자동화는 파이썬으로 돈다"만 알면, AI가 만든 결과를 이해하고 신뢰할 수 있어요.

**왜 자동화에 파이썬이 많이 쓰이나** — ① 문법이 사람 말에 가까워 AI도 가장 잘 쓰고 자료가 압도적 ② 엑셀·웹·데이터·AI를 한 언어로 ③ 무료, 윈도우 PC에 설치만 하면 됨(근태 도구의 `시작.bat`도 파이썬을 확인하고 자동 실행).

## 개념 2. 라이브러리(Library)

**한 줄 정의** — 누군가 미리 만들어 무료로 나눠준 "기능 묶음". 가져다 조립만 하면 됨.

**일상 비유** — 레고. 자동차 만들 때 바퀴를 고무부터 만들지 않죠. 잘 만들어진 "바퀴 블록"을 끼웁니다. 라이브러리가 그 블록. 표어: **"바퀴를 다시 발명하지 마라(Don't reinvent the wheel)."**

**HR 사례 연결** — 근태 도구의 `requirements.txt`가 바로 "사 온 블록 목록":
- **pandas** = 엑셀을 코드로 자유자재로 다루는 블록 (43명 × 28일 데이터 합치고 거르기)
- **openpyxl** = 엑셀 파일을 새로 만들고 색칠·수식까지 넣는 블록 (5시트 + 조건부 서식)
- **streamlit** = 웹앱 만드는 블록 (개념 3)
- **gspread / google-auth** = 구글 시트와 연결하는 블록

이 블록들이 없었다면 수만 줄이 필요했을 일이, 블록을 가져다 써서 핵심 로직 한 파일로 끝났습니다.

**왜 알아야 하나** — AI에게 "엑셀 다뤄줘" 하면 pandas를 꺼내 씁니다. 이 개념을 알면 "아, 표 다루는 블록이구나"가 보이고, "이거 직접 만들어야 하나요?"라는 겁이 줄어듭니다 — 거의 다 누가 이미 만들어 놨어요.

## 개념 3. 스트림릿(Streamlit)

**한 줄 정의** — 파이썬 몇 줄로 버튼·파일 업로드·표가 있는 "진짜 웹앱"을 만들어주는 라이브러리.

**일상 비유** — 보통 웹사이트는 디자인(HTML/CSS)·동작(JavaScript)·서버까지 "3개 국어"가 필요합니다. 스트림릿은 "파이썬 한 언어만 알면 나머지는 내가 예쁜 화면으로 바꿔줄게"라는 통역사 겸 인테리어 업자.

**HR 사례 연결** — 근태 도구가 스트림릿으로 만들어졌습니다. `app.py`를 보면:
- `st.metric("활성 인원", "43 명")` → 화면에 통계 카드
- `st.file_uploader(...)` → 엑셀 드래그앤드롭 업로드 칸
- `st.button("근태정리 생성")` → 누르면 자동화가 도는 버튼
- `st.dataframe(...)` → 결과 미리보기 표

HR 담당자는 검은 코드 화면을 볼 일이 없습니다. `시작.bat` 더블클릭 → 브라우저에 깔끔한 한글 웹앱이 뜨고, 평소 웹사이트처럼 클릭만.

**왜 비개발자 도구로 좋은가** — ① 결과가 "웹앱"이라 **동료에게 URL만 주면 같이 씀** ② 파이썬만 알면 됨(진입장벽 최저) ③ "버튼 색 바꿔줘"를 AI에게 시키면 즉시 반영.

**왜 알아야 하나** — "나만 쓰는 스크립트"로 끝낼 것이냐, "팀 전체가 클릭으로 쓰는 도구"로 만들 것이냐의 갈림길이 스트림릿. 비개발자가 "도구를 만드는 사람"이 되는 가장 빠른 길.

## 개념 4. 바이브코딩(Vibe Coding)

**한 줄 정의** — 코드를 직접 타이핑하지 않고, 자연어(평소 말)로 AI에게 설명해 코드를 만들게 하는 방식.

**일상 비유** — 예전엔 집을 지으려면 내가 벽돌을 직접 쌓았어요(직접 코딩). 바이브코딩은 유능한 시공팀에게 "여기 부엌, 창은 남향, 마루는 원목"이라고 **말로 지시**하는 것. 나는 "무엇을 원하는지"에 집중, "어떻게 짓는지"는 AI가.

**HR 사례 연결** — 강사는 파이썬을 한 줄도 외워 치지 않았습니다. "원티드랑 FLEX 엑셀을 합쳐 초과근무를 계산하는 웹앱 만들어줘", "박세진은 두 행이 있는데 WORKING만 써줘" 같은 **말**로 지시했고, Claude Code가 코드를 써냈습니다.

**장점** — ① 코딩 몰라도 도구를 만든다 ② 압도적으로 빠르다 ③ 업무를 가장 잘 아는 현업이 직접 만든다(요구 왜곡 적음).

**함정 — "그냥 만들어줘"의 위험** (반드시 짚기)
- **막연하면 엉뚱한 게 나온다**: "근태정리 도구 만들어줘"만 하면 AI는 우리 회사의 소정시간·FLEX 전환 같은 걸 몰라 일반 껍데기만.
- **그럴듯한데 틀린 결과**: 우리만의 규칙(한도 위반, 이중 카운트 방지)을 모르면, 보기엔 멀쩡한데 숫자가 틀린 엑셀을 **자신 있게** 내놓음.
- **블랙박스화**: 내용 모른 채 "되니까 OK" 하면 틀려도 못 알아챔.
→ 그래서 다음 개념(PRD·맥락·검증)이 **안전벨트**.

**왜 알아야 하나** — 오늘 실습에서 여러분이 실제로 할 행위가 이것. "코딩 배우기"가 아니라 **"AI에게 잘 시키기"**가 핵심 역량이 된 패러다임 전환.

## 개념 5. PRD(요구사항 명세) · 맥락 주기 · 검증

**한 줄 정의**
- **PRD**: 일 시키기 전에 "무엇을·왜·어떤 규칙으로" 정리한 주문서
- **맥락 주기**: 우리 회사·업무 배경을 문서로 AI에게 미리 쥐여주기(예: `CLAUDE.md`)
- **검증**: AI 결과가 진짜 맞는지 사람이 확인

**일상 비유** — 인테리어 업자에게 "예쁘게 해주세요"만 하면 내 취향과 동떨어진 집이. 반면 평면도·예산·"아이 둘, 콘센트는 여기"까지 적은 주문서(PRD) + 우리 집 사진(맥락)을 주면 결과가 확 달라집니다. 끝나면 하자 점검(검증)을 하죠. **막연한 한마디 vs 잘 쓴 주문서 — 그 차이가 바이브코딩 품질을 가른다.**

**HR 사례 연결** (강사의 진짜 무기)
- **맥락 = `01_CONTEXT.md`, `CLAUDE.md`**: 43명 명단, 소정시간, 시스템 2개 구조, 까다로운 엣지케이스 10개(박세진 두 행, 배민주 17h vs 40h, 가타하라 이중 카운트 등)를 전부 적어 AI에게 줌.
- **PRD = `02_PRD.md`**: "5시간→30분", "코딩 모르는 사람이 쓴다", "정답 엑셀과 100% 똑같이" 같은 목표·기능·비목표를 명시.
- **검증 = 12개 자동 검증 + 수기 대조**: 결과를 낼 때마다 "인원 수·합계·재계산 일치"를 점검하고 한도 위반을 화면에 띄움. 그리고 수기 최종본과 100% 대조 후 채택.

**검증의 중요성** — AI는 틀려도 당당합니다. HR·급여·근태처럼 숫자가 틀리면 사람한테 피해가 가는 업무는 **"AI 결과는 일단 의심하고 검증한다"**가 철칙. 자동화의 마지막 단계는 항상 사람의 확인.

**왜 알아야 하나** — 오늘의 진짜 한 줄: **"AI 자동화의 성패는 코딩 실력이 아니라, 얼마나 잘 설명(PRD·맥락)하고 얼마나 잘 검증하느냐."** 비개발자가 가장 잘하는 영역 — 업무는 여러분이 제일 잘 아니까요.

## 개념 연결 다이어그램 (슬라이드용)

```
 ① 내 업무 / 페인포인트
    (매월 근태정리 4~6시간 수작업, 실수 위험)
            │
            ▼
 ② PRD + 맥락 문서로 정리해서 AI에게 설명          ← 비개발자가 하는 일 (사람)
    (02_PRD.md "5h→30분, 코딩몰라도 사용"
     01_CONTEXT.md / CLAUDE.md "43명·시스템2개·엣지10개")
            │  ← 바이브코딩(자연어 지시)
            ▼
 ③ Claude Code 가 코드를 만든다                    ← AI가 하는 일 (로봇)
   ┌──────────────────────────────────────────┐
   │  파이썬(언어)                              │
   │   ├─ pandas    → 엑셀 데이터 합치기         │
   │   ├─ openpyxl  → 결과 엑셀 만들기           │   ← 라이브러리(레고 블록)
   │   └─ streamlit → 웹앱(버튼·업로드·표)       │
   └──────────────────────────────────────────┘
            │
            ▼
 ④ 웹앱 결과물  (시작.bat 더블클릭 → 브라우저)
    업로드 → 버튼 → 30초 → 5시트 엑셀
            │
            ▼
 ⑤ 검증 (사람 + 자동 12개 항목)                    ← 비개발자가 하는 일 (사람)
    "합계 일치? 한도 위반?"  →  통과하면 팀에 공유
            │
            └────── (고칠 점은 대화로 다시 ②로) ──────┘
```

**한 줄 캡션**: "내 업무 → PRD·맥락 → (Claude Code + 파이썬·라이브러리·스트림릿) → 웹앱 → 검증.
**비개발자가 맡는 칸 = ①②⑤, AI가 맡는 칸 = ③.** 여러분이 잘해야 할 건 ②(설명)와 ⑤(검증)입니다."
